### 1.1 Pre-Week-5 Cleanup Status

#### 1.1.1 `etabs.rs` is a stub — guards reference `EtabsOpen` but no implementation exists

`guards.rs` has full `EtabsOpen` guard logic. The CLI wires `ext etabs open` in
`agents.md`. But `ext-api/src/etabs.rs` is just a comment stub. This means any
CLI command that calls `etabs_open()` will fail to compile.

**Fix:** `etabs.rs` must be fully implemented in Week 5–6 (this spec covers it).

#### 1.1.2 `current_branch_name()` deduplication — resolved before Week 5–6

`commit.rs`, `switch.rs`, `checkout.rs`, `stash.rs`, and `diff.rs` all
independently inline the same `git rev-parse --abbrev-ref HEAD` call. This
violates the DRY rule and makes the branch-name logic fragile if git behavior
changes.

```rust
// ✗ Duplicated in 4 places — fragile
fn current_branch_name(ext_dir: &std::path::Path) -> String {
    std::process::Command::new("git")
        .args(["rev-parse", "--abbrev-ref", "HEAD"])
        ...
}
```

This cleanup is now part of the baseline. The shared helper lives in
`ext-core::vcs::current_branch(repo) -> Result<String>`, and the silent
fallback-to-`"main"` behaviour has been removed from the VCS/API call sites.

#### 1.1.3 `commit.rs` finalize-manifest double-commit — resolved before Week 5–6

Looking at lines 177–189 of `commit.rs`:

```rust
// Step 10: backfill git_commit_hash and rewrite manifest
manifest.git_commit_hash = Some(git_hash.clone());
manifest.write_to(&version_dir)?;
git_add(&ext_dir, &[&manifest_path])?;
git_commit(                            // ← second commit, always
    &ext_dir,
    &format!("ext: finalize manifest {}", version_id),
    ...
)?;
```

Every `ext commit` used to create **two** git commits — one user-visible and one
`ext: finalize manifest vN`. That doubled `git log` length and slowed down
`next_version_id`, which walks git history.

This cleanup is now part of the baseline. Normal commits amend in place instead
of creating a second internal finalize commit. One subtle constraint remains
important for Week 5–6 work: a commit cannot stably contain its own final
amended hash without becoming self-referential, so user-facing reads should
resolve the visible commit hash from git history rather than assuming the
in-tree `manifest.json` can always store the final amended hash directly.

```rust
// Step 9: commit
let hash = git_commit(...)?;

// Step 10: backfill hash, amend the commit in-place (no second commit)
manifest.git_commit_hash = Some(hash.clone());
manifest.write_to(&version_dir)?;
git_add(&ext_dir, &[&manifest_path])?;
// git_commit_amend instead of a new commit:
run_git(&ext_dir, &["commit", "--amend", "--no-edit"])?;
```

### 1.2 Design Issues (address during Week 5–6)

#### 1.2.1 `state.rs` does not resolve `ANALYZED` or `LOCKED`

`ext_core::state::resolve()` returns only 7 states. `ANALYZED` and `LOCKED`
are defined in the enum and referenced in guards, but the resolver never
returns them:

```rust
// state.rs - current resolver
pub fn resolve(input: ResolveInput) -> WorkingFileStatus {
    // Returns: Missing, OpenClean, OpenModified, Orphaned, Untracked, Modified, Clean
    // NEVER returns: Analyzed or Locked ← correctness gap
}
```

Currently `guards.rs` tests `Analyzed` and `Locked` in the matrix, and some
tests pass these states explicitly — but the resolver never produces them from
real disk state. Week 5–6 must wire sidecar `get-status` to distinguish
`ANALYZED` / `LOCKED` from `CLEAN`.

**Spec §4 below covers the full fix.**

#### 1.2.2 `anyhow` used in `ext-core` — should be `thiserror`

Per Apollo Rust best practices §4.3 and §4.4: `anyhow` is for binaries,
`thiserror` is for libraries. `ext-core` is a library crate but uses `anyhow`
throughout. `ext-error` exists precisely to hold `EtabsError` variants, but
the domain modules (`branch`, `stash`, `version`, `vcs`) return `anyhow::Error`
instead of typed `EtabsError` variants.

This is a medium-term issue — fixing it now would require touching every
`bail!` call across the domain. Acceptable deferral: document a ticket and
address in a future refactor sprint after Phase 1 ships. For now, add a
`#![allow(clippy::all)]` suppression comment at the top of each `ext-core`
module with a link to the tracking issue rather than leaving it silent.

**Immediate action:** Add `// TODO(#issue): migrate to thiserror for typed errors`
comments at the top of each affected module. Do not fix during Week 5–6 to
avoid scope creep.

#### 1.2.3 `analyze.rs` — `resolve_sidecar_units` error message lost at `CommitOptions::analyze`

In `commit.rs`, if `analyze = true` and `resolve_sidecar_units` fails, the
error surfaces as a generic anyhow chain. The user gets a cryptic message
instead of "Unknown unit preset 'xyz'". This is fine for now but should be
noted for Week 7–8 when `--analyze` becomes the primary path.

#### 1.2.4 `branch::list()` filesystem scan inconsistency — resolved before Week 5–6

`find_latest_version_id()` in `branch/mod.rs` scans the filesystem. But per
the spec §2.3: "The git log is the source of truth for committed versions." If
there is a partial `vN/` folder from a crash that cleanup didn't catch, the
filesystem scan may return a wrong latest version in the branch list.

This cleanup is now part of the baseline. Branch metadata reads from git-backed
version history instead of manifest directory scans, which keeps branch list
output aligned with `next_version_id`.

### 3. State Machine Extension

#### 3.1 `ANALYZED` and `LOCKED` state resolution

The current `resolve()` in `ext-core/src/state.rs` never produces `ANALYZED`
or `LOCKED`. These states require a sidecar call to detect (`is_analyzed`,
`is_locked` from `get-status`). The pure resolver cannot be extended without
breaking the "no I/O" contract of `ext-core::state`.

**Resolution strategy:** Two-tier resolution.

Tier 1 — fast path (no sidecar, used by all commands on entry):

```rust
// Same as current: returns one of 7 states
// {Missing, OpenClean, OpenModified, Orphaned, Untracked, Modified, Clean}
pub fn resolve(input: ResolveInput) -> WorkingFileStatus { ... }
```

Tier 2 — full path (sidecar call, used by `ext status --verbose` and
`ext etabs status`):

```rust
// New — in ext-api/src/status.rs (NOT in ext-core — sidecar is ext-api concern)
pub async fn resolve_with_sidecar(
    fast_status: WorkingFileStatus,
    sidecar: Option<&SidecarClient>,
    working_file: &Path,
) -> WorkingFileStatus {
    // Only called when fast_status == Clean (ANALYZED/LOCKED look like CLEAN
    // from mtime alone — ETABS is closed, file exists, mtime unchanged).
    if fast_status != WorkingFileStatus::Clean {
        return fast_status;
    }
    let Some(sidecar) = sidecar else {
        return fast_status;
    };
    match sidecar.get_status().await {
        Ok(data) if data.is_locked == Some(true) => WorkingFileStatus::Locked,
        Ok(data) if data.is_analyzed == Some(true) => WorkingFileStatus::Analyzed,
        _ => fast_status,
    }
}
```

**Where `resolve_with_sidecar` is called:**
- `ext etabs status` — always (it is the dedicated ETABS inquiry command)
- `ext status --verbose` — already does a sidecar call; fold the resolution in
- `ext etabs open` — must call it to block correctly when state is `LOCKED`
- `ext etabs close` — must know if model is `ANALYZED` to choose close mode
- `ext etabs unlock` — must confirm state is `LOCKED` before proceeding

**All other commands** use the fast path only. This keeps startup fast.

#### 3.2 Updated `ResolveInput` (no change to existing struct)

`ResolveInput` stays as-is. The fast resolver stays pure. `resolve_with_sidecar`
is an async ext-api function — no core dependency.

#### 3.3 State guards — new commands to add to `Command` enum

```rust
// ext-api/src/guards.rs — additions

    // ... existing variants ...
    EtabsClose,
    EtabsUnlock,
    EtabsRecover,

```

Permission matrix rows to add (from `workflow.md §15`):

```rust

(EtabsClose, _) => Block("✗ ETABS is not running\n  Nothing to close".into()),
```

### 4. `ext etabs open` — Full Implementation

#### 4.1 Function signature and return type

```rust
// ext-api/src/etabs.rs

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct EtabsOpenResult {
    pub opened_file: PathBuf,
    pub pid: u32,
    pub is_snapshot: bool,   // true when opening a vN/model.edb (not working file)
    pub warning: Option<String>,
}

pub async fn etabs_open(ctx: &AppContext, version_ref: Option<&str>) -> Result<EtabsOpenResult>
```

#### 4.2 Sequence

```
1. Load state + fast resolve status
2. check_state_guard(EtabsOpen, status) → Block on: OpenClean, OpenModified, Missing, Orphaned
3. resolve_with_sidecar(status, ctx.sidecar.as_ref(), working_file).await
   → if LOCKED: Block("✗ Model is locked\n  Run: ext etabs unlock before opening")
4. Determine target file:
   a. version_ref is None     → target = branch/working/model.edb (is_snapshot = false)
   b. version_ref is Some(r)  → resolve r to vN/model.edb (is_snapshot = true)
                                  → warn user: "Opening a snapshot — changes will be discarded"
5. sidecar.open_model(&target, false, false).await?   // visible, not hidden
6. Update state.json:
   wf.etabs_pid = Some(pid_from_sidecar_response)
   wf.last_known_mtime = current mtime of target file
   wf.status = WorkingFileStatus::OpenClean
   wf.status_changed_at = Utc::now()
7. Return EtabsOpenResult { opened_file: target, pid, is_snapshot, warning }
```

**Error handling:**
- Sidecar spawn fails → propagate with context "Failed to launch ETABS"
- Target file does not exist → `bail!("✗ Working file missing\n  Run: ext checkout vN")`

### 5. `ext etabs close` — Close Model / Keep ETABS Running

#### 5.1 `CloseMode` — encodes user intent

```rust
pub enum CloseMode {
    /// Prompt if OPEN_MODIFIED (default interactive behaviour).
    Interactive,
    /// Save before close without prompting.
    Save,
    /// Close without saving, discarding changes.
    NoSave,
}
```

#### 5.2 `EtabsCloseConflict` — for the CLI prompt

```rust
/// Returned when mode is Interactive and working file is OPEN_MODIFIED.
#[derive(Debug)]
pub struct EtabsCloseConflict {
    pub pid: u32,
    pub open_file: PathBuf,
}

impl std::fmt::Display for EtabsCloseConflict {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        write!(f,
            "⚠ ETABS has unsaved changes\n  \
             File: {}\n  \
             [s] Save and close  [d] Discard changes  [x] Cancel",
            self.open_file.display())
    }
}
impl std::error::Error for EtabsCloseConflict {}
```

#### 5.3 `EtabsCloseResult`

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct EtabsCloseResult {
    pub saved: bool,
    pub arrival_status: WorkingFileStatus,   // MODIFIED or ANALYZED/LOCKED or CLEAN
    pub warning: Option<String>,
}
```

#### 5.4 Sequence

```
1. Load state + fast resolve status
2. check_state_guard(EtabsClose, status)
   → Block if not (OpenClean | OpenModified | Analyzed | Locked)
3. Determine actual save intent:
   a. status == OpenClean → close without save (nothing changed)
   b. status == OpenModified, mode == Interactive → return Err(EtabsCloseConflict)
   b. status == OpenModified, mode == Save        → save = true
   c. status == OpenModified, mode == NoSave      → save = false
   d. status == Analyzed | Locked → close without save (changes already embedded)
4. sidecar.close_model(save).await?    // clears workspace, ETABS process stays alive
5. Clear state.json:
   wf.etabs_pid = None
   wf.status_changed_at = Utc::now()
6. Resolve arrival status with sidecar (Analyzed / Locked / Modified / Clean)
7. Update wf.status = arrival_status
8. Update wf.last_known_mtime = current mtime
9. save state
10. Return EtabsCloseResult { saved, arrival_status, warning }
```

**Post-close arrival status resolution:**

Important semantics: this command closes the current model/workspace only. It is
not an "exit ETABS" command, and the ETABS process may still be running on a
blank model after the close completes.

After the model closes, the sidecar no longer points at the saved working file,
so `Analyzed` / `Locked` must be persisted from the pre-close sidecar state. The
mtime check still distinguishes `Modified` from `Clean` when no analysis state
was detected.

```rust
// Arrival status after close
let mtime_after = mtime(&working_file);
let post_status = if saved {
    // File was saved — check if analysis lock was set
    resolve_with_sidecar(
        resolve(ResolveInput { ... mtime comparisons ... }),
        ctx.sidecar.as_ref(),
        &working_file,
    ).await
} else {
    // No save — recompute from mtime
    resolve(ResolveInput { ... })
};
```

### 8. `ext etabs recover` — Full Implementation

#### 8.1 `RecoveryChoice` — user decision enum

```rust
pub enum RecoveryChoice {
    /// Keep the post-crash changes as MODIFIED.
    KeepChanges,
    /// Restore from the last committed version (discard post-crash changes).
    RestoreFromVersion,
}
```

#### 8.2 `EtabsRecoverConflict` — returned for CLI prompt

```rust
#[derive(Debug)]
pub struct EtabsRecoverConflict {
    pub crashed_pid: u32,
    pub working_file: PathBuf,
    pub file_was_modified: bool,
    pub last_version: Option<String>,
    pub minutes_since_crash: Option<u64>,
}

impl std::fmt::Display for EtabsRecoverConflict {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        write!(f,
            "⚠ ETABS closed unexpectedly (PID: {})\n  \
             File modified: {}\n  \
             Last version: {}\n  \
             [k] Keep changes (mark MODIFIED)\n  \
             [r] Restore from {} (discard post-crash changes)",
            self.crashed_pid,
            if self.file_was_modified { "Yes" } else { "No" },
            self.last_version.as_deref().unwrap_or("none"),
            self.last_version.as_deref().unwrap_or("last version"),
        )
    }
}
impl std::error::Error for EtabsRecoverConflict {}
```

#### 8.3 `EtabsRecoverResult`

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct EtabsRecoverResult {
    pub choice: String,          // "kept" or "restored"
    pub arrival_status: WorkingFileStatus,
    pub working_file: PathBuf,
}
```

#### 8.4 Sequence (two-phase like checkout)

```
Phase 1 — detect (choice: None):
1. Load state
2. Fast resolve: must be ORPHANED else Block
3. Check PID is dead (OS process check — must re-verify, not trust state.json)
4. file_was_modified = current_mtime > last_known_mtime
5. Return Err(EtabsRecoverConflict { crashed_pid, working_file, file_was_modified, last_version, ... })

Phase 2 — execute (choice: Some(RecoveryChoice)):
6. clear state.json: wf.etabs_pid = None
7. If KeepChanges:
   wf.status = WorkingFileStatus::Modified
   wf.status_changed_at = now
8. If RestoreFromVersion:
   last_version = state.working_file.based_on_version
   IF last_version is None → bail("No committed version to restore from")
   src = ext_dir.join(&branch).join(last_version).join("model.edb")
   IF !src.exists() → bail("Snapshot missing: {last_version}")
   check_disk_space(&src, working_dir)?
   atomic_copy(&src, &working_file)?
   wf.based_on_version = Some(last_version)
   wf.last_known_mtime = current mtime of restored file
   wf.status = WorkingFileStatus::Clean
9. save state
10. Return EtabsRecoverResult { choice, arrival_status, working_file }
```

**Two-phase usage in CLI:**

```rust
// CLI handler (ext etabs recover)
match ext_api::etabs_recover(ctx, None).await {
    Err(e) if let Some(c) = e.downcast_ref::<EtabsRecoverConflict>() => {
        let choice = tui::prompt_recover_conflict(c)?;
        ext_api::etabs_recover(ctx, Some(choice)).await?
    }
    other => other?,
}
```

### 9. `ext etabs validate` — Deferred Until Sidecar Support Exists

This command is intentionally deferred. The current ETABS.CLI contract does not
expose a `validate-file` command yet, so Part A must not invent a Rust API
surface that the sidecar cannot actually satisfy.

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct EtabsValidateResult {
    pub file: PathBuf,
    pub is_valid: bool,
    pub etabs_version: Option<String>,
    pub error_message: Option<String>,
}

pub async fn etabs_validate(ctx: &AppContext, file: &Path) -> Result<EtabsValidateResult> {
    let sidecar = ctx.require_sidecar()?;
    let data = sidecar.validate_file(file).await?;
    Ok(EtabsValidateResult {
        file: file.to_path_buf(),
```

### 10. `ext analyze <version>` — Standalone Command

#### 10.1 Distinction from `ext commit --analyze`

- `ext commit --analyze` commits the working file first, then runs analysis on
  the new snapshot.
- `ext analyze <version>` runs analysis on an already-committed snapshot without
  creating a new version.
- Both delegate to `analyze::analyze_snapshot()` in `ext-api/src/analyze.rs`.

#### 10.2 Signature and return type

```rust
#[derive(Debug, Clone, Default)]
pub struct AnalyzeOptions {
    /// Specific load cases to run. None = all cases.
    pub cases: Option<Vec<String>>,
    /// Force re-run even if already analyzed.
    pub force: bool,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct AnalyzeResult {
    pub version_id: String,
    pub branch: String,
    pub results_dir: PathBuf,
    pub elapsed_ms: u64,
    pub warning: Option<String>,
    pub already_analyzed: bool,
}

pub async fn analyze_version(
    ctx: &AppContext,
    version_ref: &str,
    opts: AnalyzeOptions,
) -> Result<AnalyzeResult>
```

#### 10.3 Sequence

```
1. Load state + fast resolve status
2. check_state_guard(Analyze, status):
   → Block on OpenClean, OpenModified (ETABS must be closed)
   → Allow all other states (MISSING is OK — analyze doesn't touch working file)
3. Parse version_ref → (branch, version)
   → If no branch prefix: use current branch from git
4. Locate version_dir = ext_dir.join(branch).join(version)
5. Verify version_dir/model.edb exists
6. Read manifest: check is_analyzed flag
7. If already analyzed and !opts.force:
   → Return AnalyzeResult { already_analyzed: true, warning: "Use --force to re-run" }
8. sidecar = ctx.require_sidecar()?
9. Call analyze_snapshot(ctx, &version_dir, opts.cases.as_deref()).await?
10. Finalize: update manifest.is_analyzed = true, git commit "ext: analysis results vN"
11. Return AnalyzeResult
```

```rust
/// Amend the last commit without changing its message.
pub fn git_amend(repo: &Path) -> Result<String> {
```
